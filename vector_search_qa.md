# Vector Search & Embeddings Q&A

This document compiles key questions and detailed explanations regarding how vector search, embeddings, and in-memory databases work, based on our walk-through of the codebase (particularly [step-06.js](file:///c:/Users/sharafath/Downloads/LangChain-RAG/step-by-step/step-06.js)).

## 1. Can we see/inspect the in-memory database?

**Yes.** In LangChain, the [MemoryVectorStore](file:///c:/Users/sharafath/Downloads/LangChain-RAG/node_modules/langchain/dist/vectorstores/memory.d.ts#L28-L99) stores its data in the `memoryVectors` property.

Each vector inside this array follows the `MemoryVector` structure defined in [memory.d.ts:L10-L14](file:///c:/Users/sharafath/Downloads/LangChain-RAG/node_modules/langchain/dist/vectorstores/memory.d.ts#L10-L14):
```typescript
interface MemoryVector {
    content: string;
    embedding: number[];
    metadata: Record<string, any>;
}
```

To print a formatted preview of your in-memory database contents, you can map the array and print a truncated version (to avoid dumping hundreds of float coordinates to the console):

```javascript
const dbPreview = pdfQa.db.memoryVectors.map((vec, index) => ({
  id: index + 1,
  metadata: vec.metadata,
  embeddingDimension: vec.embedding.length,
  contentSnippet: vec.content.trim().slice(0, 80) + "..."
}));

console.log("\n--- In-Memory Vector Store Contents ---");
console.dir(dbPreview, { depth: null, colors: true });
```

## 2. Why do the same page numbers (e.g. Page 3, 6, 7) appear multiple times with different similarity scores?

During document splitting ([splitDocuments()](file:///c:/Users/sharafath/Downloads/LangChain-RAG/step-by-step/step-06.js#L40-L49)), a single PDF page is divided into smaller text chunks (e.g., 1000 characters). 

* Each chunk is embedded and stored as a separate vector in the database.
* All chunks from a page retain the same page number in their `metadata` (e.g., `doc.metadata.loc.pageNumber`).
* When performing a similarity search, multiple chunks from the same page can match your query. Each chunk will have a slightly different similarity score depending on how closely its specific text matches the query.

---

## 3. What is a "Vector" and "Vector Space"? Is it like a "Space DB"?

Yes! The concept of a **"Space DB"** is a perfect way to visualize it.

* **Vector:** An array of floating-point numbers (e.g. 384 numbers for `all-minilm`) generated by the embedding AI.
* **Vector Space (Embedding Space):** A multi-dimensional mathematical space where the numbers act as coordinates. 
* The AI maps the *semantic meaning* of text into this space. Semantically similar texts are placed close to each other, while unrelated texts are far apart.

---

## 4. What does the similarity score (e.g., `SIM=0.592`) represent?

The score (calculated using *Cosine Similarity*) measures the angle between the query vector and the document vector in the vector space:
* **`1.0`**: Perfect semantic match (pointing in the exact same direction).
* **`0.0`**: Zero similarity (perpendicular/completely unrelated).
* In real-world applications, similarity scores typically range between `0.2` and `0.8`.

---

## 5. Do coordinates change when a new database is created?

It depends on the model:
* **Same model (`all-minilm`):** No. The embedding model is deterministic. Re-running the script or rebuilding the database with the same text will always output the exact same coordinates.
* **Different model:** Yes. If you switch to another model (like OpenAI or `nomic-embed-text`), the coordinates will change because the new model uses a different coordinate system and a different number of dimensions.

---

## 6. Why do we search vector-to-vector instead of string-to-string?

Traditional **string-to-string (keyword)** search matches exact letters/characters, which has several limitations:
1. **Misses synonyms:** A query for `"close the app"` won't match a document saying `"exit the application"`.
2. **Lacks context:** It can't differentiate between the word "bank" in a financial context versus a riverbank context.
3. **Fails with typos/grammar:** Grammatical differences (like plurals) or typos can break exact string matches.

**Vector-to-vector (semantic)** search maps the *concepts* behind the words, finding correct matches even if the query uses entirely different words than the document.

---

## 7. Does it run the AI every time a search or match is executed?

Only to translate text into coordinates. The comparison itself is pure math.

1. **Write Phase (Database Creation):** The **AI model** embeds the text chunks once and saves the vectors in the database.
2. **Read Phase (Search):** The **AI model** embeds the query string *once* to get its query vector.
3. **Match Phase (Retrieval):** The database performs **simple geometry (no AI)** to find which stored vectors are mathematically closest to the query vector. This is fast and lightweight.

---

## 8. How does a small query (e.g., 3 words) match a large paragraph (e.g., 200 words)?

* The embedding AI acts as a summarizer that compresses text into a **"semantic fingerprint"** of a fixed size (e.g., exactly 384 dimensions), regardless of the input length.
* Since the 3-word query and the 200-word paragraph discuss the same core concepts, their final coordinates end up very close to each other in the vector space.

---

## 9. Does a single chunk contain multiple vectors?

**No. One chunk = Exactly one vector.** 
The model processes the entire chunk as a single unit and outputs one coordinate array. If your database has 40 chunks, it contains exactly 40 vectors.

---

## 10. Does a match return only the word or the entire paragraph?

It returns the **entire text chunk (the full paragraph)**. The returned `Document` object contains the `pageContent` property, which holds the complete text of the matched chunk, and the `metadata` property with its source details.
